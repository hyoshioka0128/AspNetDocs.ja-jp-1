# <a name="what-is-web-forms"></a>Web フォームとは

ASP.NET Web Forms は ASP.NET web アプリケーションフレームワークの一部であり、 [Visual Studio](https://www.asp.net/downloads)に含まれています。 ASP.NET web アプリケーションの作成に使用できる4つのプログラミングモデルの1つであり、他のモデルは ASP.NET MVC、ASP.NET Web ページ、および ASP.NET のシングルページアプリケーションです。

Web フォームは、ユーザーがブラウザーを使用して要求するページです。 これらのページは、HTML、クライアントスクリプト、サーバーコントロール、およびサーバーコードの組み合わせを使用して記述できます。 ユーザーがページを要求すると、フレームワークによってコンパイルされ、サーバー上で実行されます。その後、フレームワークは、ブラウザーがレンダリングできる HTML マークアップを生成します。 ASP.NET Web フォームページでは、ブラウザーまたはクライアントデバイスでユーザーに情報が表示されます。

Visual Studio を使用して、ASP.NET Web フォームを作成することができます。 Visual Studio 統合開発環境 (IDE) を使用すると、サーバーコントロールをドラッグアンドドロップして Web フォームページをレイアウトできます。 その後、ページまたはページ自体のコントロールに対して、プロパティ、メソッド、およびイベントを簡単に設定できます。 これらのプロパティ、メソッド、およびイベントは、web ページの動作、ルックアンドフィールなどを定義するために使用されます。 ページのロジックを処理するサーバーコードを記述するには、Visual Basic やC#のような .net 言語を使用できます。

> [!NOTE] 
> 
> ASP.NET と Visual Studio のドキュメントは、いくつかのバージョンにまたがります。 以前のバージョンの機能を強調表示するトピックは、最新バージョンを使用している現在のタスクとシナリオで役立つ場合があります。

**ASP.NET Web フォームは次のとおりです。** 

- サーバー上で実行されるコードがブラウザーまたはクライアントデバイスに Web ページ出力を動的に生成する Microsoft ASP.NET テクノロジに基づいています。
- 任意のブラウザーやモバイルデバイスと互換性があります。 ASP.NET Web ページでは、スタイルやレイアウトなどの機能に適したブラウザーに準拠した HTML が自動的にレンダリングされます。
- Microsoft Visual Basic や Microsoft Visual C#など、.net 共通言語ランタイムでサポートされている任意の言語と互換性があります。
- Microsoft .NET Framework 上に構築されています。 これにより、管理された環境、タイプセーフ、継承など、フレームワークのすべての利点が得られます。
- ユーザー作成のコントロールとサードパーティのコントロールを追加できるため、柔軟性があります。

**ASP.NET Web フォームプラン:** 

- アプリケーションロジックから HTML とその他の UI コードを分離する。
- データアクセスなど、一般的なタスクのためのサーバーコントロールの豊富なスイート。
- 優れたツールサポートを備えた強力なデータバインディング。
- ブラウザーで実行されるクライアント側スクリプトのサポート。
- ルーティング、セキュリティ、パフォーマンス、国際化、テスト、デバッグ、エラー処理、状態管理など、さまざまな機能がサポートされています。

## <a name="aspnet-web-forms-helps-you-overcome-challenges"></a>ASP.NET Web フォームは、課題を克服するのに役立ちます。

Web アプリケーションプログラミングでは、従来のクライアントベースのアプリケーションをプログラミングするときに通常は発生しない課題が示されています。 次のような課題があります。

- **豊富な Web ユーザーインターフェイスを実装**することは、基本的な HTML 機能を使用してユーザーインターフェイスを設計および実装するのが困難で面倒な場合があります。特に、ページが複雑なレイアウト、大量の動的なコンテンツ、およびすべての機能を備えたユーザー対話型のオブジェクトである場合に便利です。
- **クライアントとサーバーの分離**-Web アプリケーションでは、クライアント (ブラウザー) とサーバーは異なるコンピューター (および異なるオペレーティングシステムでも) で実行されることが多いプログラムです。 このため、アプリケーションの2つの部分では、ごくわずかな情報を共有しています。これらは通信できますが、通常、単純な情報の小さなまとまりだけを交換します。
- **ステートレス実行**-Web サーバーがページの要求を受信すると、ページが検索されて処理され、ブラウザーに送信された後、すべてのページ情報が破棄されます。 ユーザーが同じページを再度要求した場合、サーバーはシーケンス全体を繰り返し、ページをゼロから再処理します。 別の方法として、処理したページのメモリがサーバーに存在しないことがあります。ページはステートレスです。 そのため、アプリケーションでページに関する情報を保持する必要がある場合、そのステートレスな性質が問題になる可能性があります。
- **不明なクライアント機能**-多くの場合、Web アプリケーションはさまざまなブラウザーを使用して多数のユーザーにアクセスできます。 ブラウザーにはさまざまな機能があるため、アプリケーションを作成することは難しく、すべてのアプリケーションで同様に動作します。
- **データアクセスの複雑さ**-従来の Web アプリケーションでのデータソースの読み取りと書き込みは複雑で、リソースが大量に消費される可能性があります。
- **スケーラビリティの複雑さ**-多くの場合、既存のメソッドを使用して設計された Web アプリケーションは、アプリケーションのさまざまなコンポーネント間の互換性がないため、スケーラビリティの目標を満たすことができません。 これは、多くの場合、増加サイクルが大きいアプリケーションでは一般的な障害点です。

Web アプリケーションに対してこれらの課題を満たすには、かなりの時間と労力が必要になることがあります。 ASP.NET Web フォームと ASP.NET フレームワークは、次のような課題に対処します。

- **直感的で一貫性のあるオブジェクトモデル**-ASP.NET page framework には、フォームを個別のクライアントとサーバーの部分ではなく、1つの単位として考えることができるオブジェクトモデルが用意されています。 このモデルでは、ページ要素のプロパティを設定したり、イベントに応答したりするなど、従来の Web アプリケーションよりも直感的な方法でページをプログラミングできます。 また、ASP.NET サーバーコントロールは、HTML ページの物理的な内容や、ブラウザーとサーバー間の直接やり取りからの抽象化です。 一般に、サーバーコントロールは、クライアントアプリケーションのコントロールを操作する方法と、コントロールとその内容を表示および処理するための HTML の作成方法を考慮する必要がない方法で使用できます。
- **イベントドリブンプログラミングモデル**-ASP.NET web Forms は、クライアントまたはサーバーで発生するイベントのイベントハンドラーを作成するための使い慣れたモデルを web アプリケーションに取り込みます。 ASP.NET page framework では、クライアント上でイベントをキャプチャしてサーバーに送信し、適切なメソッドを呼び出すための基礎となるメカニズムが、すべて自動的に非表示になるように、このモデルが抽象化されています。 その結果、イベントドリブン開発をサポートする明確で簡単に記述されたコード構造になります。
- **直感的な状態管理**-ASP.NET page framework は、ページとそのコントロールの状態を維持するタスクを自動的に処理し、アプリケーション固有の情報の状態を維持するための明示的な方法を提供します。 これは、サーバーリソースを多用せずに実行でき、cookie をブラウザーに送信したり、使用したりすることなく実装できます。
- **ブラウザーに依存しないアプリケーション**-ASP.NET page フレームワークを使用すると、サーバー上のすべてのアプリケーションロジックを作成でき、ブラウザーの違いを明示的にコーディングする必要がなくなります。 ただし、パフォーマンスの向上とクライアントエクスペリエンスの向上を実現するためにクライアント側のコードを記述することで、ブラウザー固有の機能を利用できます。
- **共通言語ランタイムサポートの .NET Framework** -ASP.NET page framework は .NET Framework に基づいて構築されているため、すべての ASP.NET アプリケーションでフレームワーク全体を使用できます。 アプリケーションは、ランタイムと互換性のある任意の言語で記述できます。 さらに、.NET Framework によって提供されるデータアクセスインフラストラクチャ (ADO.NET を含む) を使用してデータアクセスが簡略化されます。
- **スケーラブルなサーバーパフォーマンスの .NET Framework** -ASP.NET page フレームワークを使用すると、アプリケーションのロジックに複雑な変更を加えることなく、1つのプロセッサを持つ1台のコンピューターからマルチコンピューターの web ファームに、web アプリケーションを完全に拡張することができます。

## <a name="features-of-aspnet-web-forms"></a>ASP.NET Web フォームの機能

- **サーバーコントロール**-ASP.NET web サーバーコントロールは、ページが要求され、ブラウザーにマークアップをレンダリングするときに実行される ASP.NET web ページ上のオブジェクトです。 多くの Web サーバーコントロールは、ボタンやテキストボックスなどの使い慣れた HTML 要素に似ています。 その他のコントロールには、カレンダーコントロールなどの複雑な動作や、データソースへの接続とデータの表示に使用できるコントロールが含まれます。
- **マスターページ**-ASP.NET マスターページを使用すると、アプリケーションのページに対して一貫したレイアウトを作成できます。 1つのマスターページで、アプリケーション内のすべてのページ (またはページのグループ) に対して必要なルックアンドフィールと標準動作を定義します。 その後、表示するコンテンツを含む個々のコンテンツページを作成できます。 ユーザーがコンテンツページを要求すると、マスターページとマージされ、マスターページのレイアウトとコンテンツページのコンテンツを組み合わせた出力が生成されます。
- ASP.NET を使用すると、データを格納、取得、および表示するためのさまざまなオプション**が**提供されます。 ASP.NET Web フォームアプリケーションでは、データバインドコントロールを使用して、web ページの UI 要素 (テーブル、テキストボックス、ドロップダウンリストなど) でのデータの表示や入力を自動化します。
- **メンバーシップ**ASP.NET Identity は、アプリケーションによって作成されたデータベースにユーザーの資格情報を格納します。 ユーザーがログインすると、アプリケーションはデータベースを読み取って資格情報を検証します。 プロジェクトの*アカウント*フォルダーには、メンバーシップのさまざまな部分を実装するファイルが含まれています。これには、登録、ログイン、パスワードの変更、およびアクセスの承認が含まれます。 また、ASP.NET Web フォームは OAuth および OpenID もサポートしています。 これらの認証拡張機能により、ユーザーは、Facebook、Twitter、Windows Live、Google などのアカウントから既存の資格情報を使用してサイトにログインできるようになります。 既定では、このテンプレートは SQL Server Express LocalDB のインスタンスの既定のデータベース名を使用してメンバーシップデータベースを作成します。これは、Web 用の Visual Studio Express 2013 に付属する開発データベースサーバーです。
- **クライアントスクリプトとクライアントフレームワーク**-ASP.NET Web フォームページにクライアントスクリプト機能を含めることによって、ASP.NET のサーバーベースの機能を強化できます。 クライアントスクリプトを使用すると、より高度で応答性の高いユーザーインターフェイスをユーザーに提供できます。 また、ブラウザーでページが実行されているときに、クライアントスクリプトを使用して Web サーバーに非同期呼び出しを行うこともできます。
- **ルーティング**-URL ルーティングを使用すると、物理ファイルにマップされていない要求 url を受け入れるようにアプリケーションを構成できます。 要求 URL は、ユーザーがブラウザーに入力して web サイト上のページを見つけるための URL です。 ルーティングを使用して、ユーザーに意味のある Url を定義し、検索エンジンの最適化 (SEO) に役立てることができます。
- **状態管理**-ASP.NET Web フォームには、ページごとにデータを保存するためのオプションと、アプリケーション全体に適用するオプションがいくつか用意されています。
- **セキュリティ**-セキュリティで保護されたアプリケーションの開発の重要な部分は、脅威を理解することです。 マイクロソフトは、スプーフィング、改ざん、否認、情報開示、サービス拒否、権限の昇格 (STRIDE) などの脅威を分類する方法を開発しました。 ASP.NET Web フォームでは、ASP.NET Web フォームのさまざまなセキュリティ動作をカスタマイズできる機能拡張ポイントと構成オプションを追加できます。
- **パフォーマンス**-Web サイトまたはプロジェクトを成功させるには、パフォーマンスが重要な要素になることがあります。 ASP.NET Web フォームを使用すると、ページおよびサーバーコントロールの処理、状態管理、データアクセス、アプリケーションの構成と読み込み、効率的なコーディングの手法に関連するパフォーマンスを変更できます。
- ASP.NET Web フォームを使用**すると、** ブラウザーの優先言語設定に基づいて、またはユーザーが明示的に選択した言語に基づいて、コンテンツやその他のデータを取得できる web ページを作成できます。 コンテンツなどのデータはリソースと呼ばれ、そのようなデータはリソースファイルやその他のソースに格納できます。 ASP.NET Web フォームページでは、リソースからプロパティ値を取得するようにコントロールを構成します。 実行時に、リソース式はローカライズされた適切なリソースファイルのリソースに置き換えられます。
- **デバッグとエラー処理**-ASP.NET には、Web フォームアプリケーションで発生する可能性のある問題を診断するのに役立つ機能が含まれています。 デバッグとエラー処理は ASP.NET Web フォーム内で適切にサポートされているため、アプリケーションを効率的にコンパイルして実行できます。
- **デプロイとホスティング**-Visual Studio、ASP.NET、Azure、および IIS は、Web フォームアプリケーションを配置およびホストするプロセスを支援するツールを提供します。

## <a name="deciding-when-to-create-a-web-forms-application"></a>Web フォームアプリケーションを作成する場合の判断

ASP.NET Web フォームモデルまたは ASP.NET MVC フレームワークなどの別のモデルを使用して、Web アプリケーションを実装するかどうかを慎重に検討する必要があります。 MVC フレームワークは、Web フォーム モデルに取って代わるものではありません。どちらのフレームワークも Web アプリケーションに使用できます 特定の Web サイトで Web フォームモデルまたは MVC フレームワークを使用する場合は、それぞれの方法の利点を検討してください。

### <a name="advantages-of-a-web-forms-based-web-application"></a>Web フォーム ベースの Web アプリケーションの利点

Web フォーム ベースのフレームワークには、次のような利点があります。

- HTTP 経由で状態を保持するイベント モデルがサポートされます。これは、基幹業務 Web アプリケーションの開発に有用です。 Web フォーム ベースのアプリケーションでは、数多くのサーバー コントロールでサポートされているさまざまなイベントを使用できます。
- 個々のページに機能を追加するページ コントローラー パターンが使用されます。 詳細については、MSDN Web サイトの「[ページコントローラー](https://go.microsoft.com/fwlink/?LinkId=106359 "ページコントローラー") 」を参照してください。
- ビューステートまたはサーバーベースのフォームを使用します。これにより、状態情報の管理が容易になります。
- Web 開発者とデザイナーで構成される小規模なチームで、RAD (Rapid Application Development) のための多数のコンポーネントを活用する場合に適しています。
- 一般に、コンポーネント (**ページ**クラス、コントロールなど) は密接に統合されており、MVC モデルよりもコードが少なくなるため、アプリケーションの開発が複雑になることはほとんどありません。

### <a name="advantages-of-an-mvc-based-web-application"></a>MVC ベースの Web アプリケーションの利点

ASP.NET MVC フレームワークには次のような利点があります。

- アプリケーションがモデル、ビュー、およびコントローラーに分離されるため、複雑さが軽減されます。
- ビュー状態やサーバー ベースのフォームは使用されません。 このため、MVC フレームワークは、アプリケーションの動作を完全に制御する必要がある開発者に向いています。
- Web アプリケーションの要求を単一のコントローラーで処理するフロント コントローラー パターンが使用されます。 これにより、多機能なルーティング インフラストラクチャをサポートするアプリケーションを設計できます。 詳細については、MSDN Web サイトの「 [Front Controller](https://go.microsoft.com/fwlink/?LinkId=106357 "フロントコントローラー") 」を参照してください。
- テスト駆動開発 (TDD) のサポートが向上しています。
- アプリケーションの動作を高度に制御する必要がある開発者や Web デザイナーの大規模なチームによってサポートされる Web アプリケーションに適しています。
